## 二、核心难点分析

### 难点1：层级关系深度不确定

族谱是一棵**树形结构**，每个成员可能有父节点（父亲/母亲），深度不固定。
- 在关系数据库中，通常用 **"邻接表模型"**（Adjacency List）来表示：
  - 每个成员记录中存一个 `parent_id`（指向其父亲/母亲的ID）
  - 这种方式简单，但查询祖先需要**递归**

### 难点2：同名同姓不同辈分

解决方案：每个成员有唯一的 `member_id`，所有查询都基于ID而不是名字。
- 名字只是显示字段，不作为唯一标识。

### 难点3：快速查询直系后代/祖先

- **祖先查询**：用 MySQL 8.0+ 的 **递归 CTE（WITH RECURSIVE）** 自底向上追溯
- **后代查询**：同样用递归 CTE，自顶向下展开

### 难点4：两个人之间的亲缘链路

- 需要从两个人分别向上追溯祖先，找到**最近共同祖先（LCA）**
- 然后拼接两条路径形成完整链路

## 三、数据库设计

### 3.1 ER图设计

**实体：**

| 实体 | 属性 |
|------|------|
| User（用户） | user_id, username, password_hash, email, created_at |
| Genealogy（族谱） | genealogy_id, name, surname, created_by, created_at |
| Member（成员） | member_id, genealogy_id, name, gender, birth_year, death_year, bio |
| FamilyLink（血缘关系） | link_id, child_id, parent_id, relation_type（father/mother） |
| Marriage（婚姻关系） | marriage_id, member_id1, member_id2, marriage_date, divorce_date |
| Collaboration（协作） | collaboration_id, user_id, genealogy_id, role |

**联系类型：**
- User --(1:N)--> Genealogy：一个用户可创建多个族谱
- User --(M:N)--> Genealogy：通过 Collaboration 表实现多对多（邀请协作）
- Genealogy --(1:N)--> Member：一个族谱包含多个成员
- Member --(M:N)--> Member（通过 FamilyLink）：成员间的父子/母子关系
- Member --(1:1 或 1:N)--> Member（通过 Marriage）：婚姻关系

### 3.2 关系表设计（MySQL）

```sql
-- 1. 用户表
CREATE TABLE users (
    user_id INT AUTO_INCREMENT PRIMARY KEY,
    username VARCHAR(50) NOT NULL UNIQUE,
    password_hash VARCHAR(255) NOT NULL,
    email VARCHAR(100),
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP
);

-- 2. 族谱表
CREATE TABLE genealogies (
    genealogy_id INT AUTO_INCREMENT PRIMARY KEY,
    name VARCHAR(100) NOT NULL,
    surname VARCHAR(50) NOT NULL,
    created_by INT NOT NULL,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (created_by) REFERENCES users(user_id)
);

-- 3. 成员表
CREATE TABLE members (
    member_id INT AUTO_INCREMENT PRIMARY KEY,
    genealogy_id INT NOT NULL,
    name VARCHAR(100) NOT NULL,
    gender ENUM('M', 'F') NOT NULL,
    birth_year INT,
    death_year INT,
    bio TEXT,
    FOREIGN KEY (genealogy_id) REFERENCES genealogies(genealogy_id)
);

-- 4. 血缘关系表（邻接表模型）
CREATE TABLE family_links (
    link_id INT AUTO_INCREMENT PRIMARY KEY,
    child_id INT NOT NULL,
    parent_id INT NOT NULL,
    relation_type ENUM('father', 'mother') NOT NULL,
    FOREIGN KEY (child_id) REFERENCES members(member_id),
    FOREIGN KEY (parent_id) REFERENCES members(member_id),
    UNIQUE KEY unique_parent_child (child_id, parent_id)
);

-- 5. 婚姻关系表
CREATE TABLE marriages (
    marriage_id INT AUTO_INCREMENT PRIMARY KEY,
    member_id1 INT NOT NULL,
    member_id2 INT NOT NULL,
    marriage_year INT,
    divorce_year INT,
    FOREIGN KEY (member_id1) REFERENCES members(member_id),
    FOREIGN KEY (member_id2) REFERENCES members(member_id)
);

-- 6. 协作表
CREATE TABLE collaborations (
    collaboration_id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    genealogy_id INT NOT NULL,
    role ENUM('editor', 'viewer') DEFAULT 'editor',
    FOREIGN KEY (user_id) REFERENCES users(user_id),
    FOREIGN KEY (genealogy_id) REFERENCES genealogies(genealogy_id),
    UNIQUE KEY unique_collab (user_id, genealogy_id)
);
```

### 3.3 范式分析（3NF/BCNF）

- **1NF**：所有字段都是原子值（性别用ENUM，不用逗号分隔）
- **2NF**：每个非主属性完全依赖于主键（没有部分依赖）
- **3NF**：没有传递依赖。例如：
  - `members` 表中，`bio` 直接依赖于 `member_id`，不依赖于其他非主属性
  - `family_links` 中，`relation_type` 依赖于 `(child_id, parent_id)` 这个组合键
- **BCNF**：每个决定因素都是候选键。以上设计满足BCNF。

## 四、关键SQL实现

### 4.1 递归CTE — 查询某成员的所有祖先（核心！）

```sql
-- 输入：成员ID (例如 member_id = 100)
-- 输出：该成员向上追溯的所有历代祖先

WITH RECURSIVE ancestors AS (
    -- 锚点成员：从给定成员开始
    SELECT m.member_id, m.name, m.gender, m.birth_year, 
           fl.parent_id, fl.relation_type, 1 AS generation
    FROM members m
    JOIN family_links fl ON m.member_id = fl.child_id
    WHERE m.member_id = 100
    
    UNION ALL
    
    -- 递归：继续向上找父母
    SELECT m.member_id, m.name, m.gender, m.birth_year,
           fl.parent_id, fl.relation_type, a.generation + 1
    FROM ancestors a
    JOIN members m ON a.parent_id = m.member_id
    JOIN family_links fl ON m.member_id = fl.child_id
)
SELECT * FROM ancestors ORDER BY generation;
```

**底层机制：**
- MySQL 在内存中维护一个**临时工作表**
- 每次迭代取出上一轮的 `parent_id` 作为新的 `child_id` 查找
- 当 `parent_id IS NULL`（已到根节点）时停止
- 这本质上是一个**图的BFS/DFS遍历**

### 4.2 递归CTE — 查询某成员的所有后代

```sql
WITH RECURSIVE descendants AS (
    SELECT m.member_id, m.name, m.gender, m.birth_year, 1 AS generation
    FROM members m
    WHERE m.member_id = 100
    
    UNION ALL
    
    SELECT m.member_id, m.name, m.gender, m.birth_year, d.generation + 1
    FROM descendants d
    JOIN family_links fl ON d.member_id = fl.parent_id
    JOIN members m ON fl.child_id = m.member_id
)
SELECT * FROM descendants ORDER BY generation;
```

### 4.3 查询两人之间的亲缘链路

```sql
-- 查询 member_id = A 和 member_id = B 之间的亲缘链路
-- 思路：分别向上追溯到根，找到最近共同祖先(LCA)，再拼接路径

-- 步骤1：获取 A 的所有祖先路径
WITH RECURSIVE path_a AS (
    SELECT m.member_id, m.name, m.parent_id, 0 AS distance
    FROM members m WHERE m.member_id = 100
    UNION ALL
    SELECT m.member_id, m.name, fl.parent_id, pa.distance + 1
    FROM path_a pa
    JOIN family_links fl ON pa.member_id = fl.child_id
    JOIN members m ON fl.parent_id = m.member_id
),
-- 步骤2：获取 B 的所有祖先路径
path_b AS (
    SELECT m.member_id, m.name, m.parent_id, 0 AS distance
    FROM members m WHERE m.member_id = 200
    UNION ALL
    SELECT m.member_id, m.name, fl.parent_id, pb.distance + 1
    FROM path_b pb
    JOIN family_links fl ON pb.member_id = fl.child_id
    JOIN members m ON fl.parent_id = m.member_id
)
-- 步骤3：找到最近共同祖先
SELECT pa.member_id AS common_ancestor, 
       pa.distance AS dist_from_a, 
       pb.distance AS dist_from_b
FROM path_a pa
JOIN path_b pb ON pa.member_id = pb.member_id
ORDER BY (pa.distance + pb.distance) ASC
LIMIT 1;
```

### 4.4 统计查询

```sql
-- 统计平均寿命最长的一代人
WITH member_generations AS (
    SELECT m.member_id, 
           (m.death_year - m.birth_year) AS lifespan,
           -- 计算辈分：用递归CTE向上追溯到根
           0 AS gen  -- 简化示例
    FROM members m
    WHERE m.death_year IS NOT NULL AND m.birth_year IS NOT NULL
)
SELECT gen, AVG(lifespan) AS avg_lifespan
FROM member_generations
GROUP BY gen
ORDER BY avg_lifespan DESC
LIMIT 1;

-- 查询年龄超过50岁且没有配偶的男性成员
SELECT m.* 
FROM members m
LEFT JOIN marriages mar1 ON m.member_id = mar1.member_id1
LEFT JOIN marriages mar2 ON m.member_id = mar2.member2
WHERE m.gender = 'M'
  AND (YEAR(CURDATE()) - m.birth_year) > 50
  AND mar1.marriage_id IS NULL 
  AND mar2.marriage_id IS NULL;
```

## 五、索引设计

```sql
-- 模糊查询姓名（使用 B-Tree 索引，支持前缀匹配）
CREATE INDEX idx_member_name ON members(genealogy_id, name);

-- 根据父节点ID查询子节点（family_links最常用查询）
CREATE INDEX idx_family_links_child ON family_links(child_id);
CREATE INDEX idx_family_links_parent ON family_links(parent_id);

-- 递归CTE性能关键索引
CREATE INDEX idx_family_links_child_parent ON family_links(child_id, parent_id);

-- 成员表外键索引
CREATE INDEX idx_members_genealogy ON members(genealogy_id);
```

### 索引性能对比

### 7.2 调用链路（一次请求的完整路径）

以"用户点击查询祖先按钮"为例，展示一次HTTP请求在各层之间的流转：

```
用户浏览器 (HTTP GET /relations/ancestors?member_id=100)
    │
    ▼
routes/relation_routes.py          ← ① 接收请求，提取参数
    │  调用 services
    ▼
services/relation_service.py       ← ② 业务逻辑：参数校验，权限检查
    │  调用 dao
    ▼
dao/family_link_dao.py             ← ③ 执行递归CTE的SQL语句
    │  调用数据库
    ▼
MySQL 数据库                        ← ④ 返回查询结果（游标）
    │
    ▼ （结果逐层返回）
dao 将结果转为 Member 对象列表
    │
    ▼
service 做业务处理（路径拼接等）
    │
    ▼
route 将结果转为 JSON/HTML 响应
    │
    ▼
用户浏览器展示祖先树形图
```

**关键约束**：每一层只调用其直接下层，禁止跨层调用（如 route 层禁止直接写 SQL）。

### 7.3 每层代码示例（以成员模块为例）

#### models/member.py — 数据模型（对应DB层的表结构）

```python
class Member:
    """成员数据模型，对应 members 表"""
    
    def __init__(self, member_id, genealogy_id, name, gender,
                 birth_year=None, death_year=None, bio=None):
        self.member_id = member_id
        self.genealogy_id = genealogy_id
        self.name = name
        self.gender = gender          # 'M' 或 'F'
        self.birth_year = birth_year
        self.death_year = death_year
        self.bio = bio
    
    def to_dict(self):
        """转为字典，便于JSON序列化"""
        return {
            'member_id': self.member_id,
            'genealogy_id': self.genealogy_id,
            'name': self.name,
            'gender': self.gender,
            'birth_year': self.birth_year,
            'death_year': self.death_year,
            'bio': self.bio
        }
    
    def age(self):
        """计算年龄（如果还在世）"""
        if self.death_year:
            return self.death_year - self.birth_year
        from datetime import datetime
        return datetime.now().year - self.birth_year
```

#### dao/member_dao.py — 数据访问层（只写SQL，不做业务判断）

```python
"""
数据访问层：只负责执行SQL，不包含任何业务逻辑。
每个函数对应一个或一组SQL操作。
"""

def get_member_by_id(conn, member_id):
    """根据ID查询单个成员"""
    cursor = conn.cursor(dictionary=True)
    cursor.execute("SELECT * FROM members WHERE member_id = %s", (member_id,))
    return cursor.fetchone()          # 返回 dict 或 None

def search_by_name(conn, genealogy_id, keyword):
    """根据姓名模糊搜索（前缀匹配）"""
    cursor = conn.cursor(dictionary=True)
    cursor.execute(
        "SELECT * FROM members WHERE genealogy_id = %s AND name LIKE %s",
        (genealogy_id, f"%{keyword}%")
    )
    return cursor.fetchall()          # 返回 list[dict]

def get_members_by_genealogy(conn, genealogy_id):
    """获取某族谱的全部成员"""
    cursor = conn.cursor(dictionary=True)
    cursor.execute(
        "SELECT * FROM members WHERE genealogy_id = %s ORDER BY birth_year",
        (genealogy_id,)
    )
    return cursor.fetchall()

def insert_member(conn, genealogy_id, name, gender, birth_year, death_year, bio):
    """插入新成员"""
    cursor = conn.cursor()
    cursor.execute(
        "INSERT INTO members (genealogy_id, name, gender, birth_year, death_year, bio) "
        "VALUES (%s, %s, %s, %s, %s, %s)",
        (genealogy_id, name, gender, birth_year, death_year, bio)
    )
    conn.commit()
    return cursor.lastrowid           # 返回新插入的 member_id
```

#### services/member_service.py — 业务逻辑层（调用DAO，做业务判断）

```python
"""
业务逻辑层：负责业务规则校验、权限控制、流程编排。
调用 dao 层获取数据，不做直接的SQL操作。
"""
from dao.member_dao import get_member_by_id, search_by_name, insert_member

def get_member(conn, member_id, current_user_id):
    """
    获取成员信息（含权限校验）
    返回: (member_dict, error_message)
    """
    # 1. 调用 DAO 获取数据
    member = get_member_by_id(conn, member_id)
    if not member:
        return None, "成员不存在"
    
    # 2. 业务逻辑：校验当前用户是否有权查看
    if not _has_access(conn, current_user_id, member['genealogy_id']):
        return None, "无权访问该族谱"
    
    return member, None

def search_members(conn, genealogy_id, keyword, current_user_id):
    """模糊搜索成员（含权限校验）"""
    if not _has_access(conn, current_user_id, genealogy_id):
        return [], "无权访问该族谱"
    
    results = search_by_name(conn, genealogy_id, keyword)
    return results, None

def create_member(conn, genealogy_id, name, gender, birth_year, current_user_id):
    """创建新成员（含业务规则校验）"""
    # 权限校验
    if not _has_edit_access(conn, current_user_id, genealogy_id):
        return None, "无编辑权限"
    
    # 数据校验：出生年份合理性
    if birth_year and (birth_year < 1000 or birth_year > 2100):
        return None, "出生年份不合理"
    
    member_id = insert_member(conn, genealogy_id, name, gender, birth_year, None, None)
    return {'member_id': member_id}, None

def _has_access(conn, user_id, genealogy_id):
    """判断用户是否有权访问某族谱（创建者或协作者）"""
    cursor = conn.cursor()
    cursor.execute("""
        SELECT 1 FROM genealogies WHERE genealogy_id = %s AND created_by = %s
        UNION
        SELECT 1 FROM collaborations WHERE genealogy_id = %s AND user_id = %s
    """, (genealogy_id, user_id, genealogy_id, user_id))
    return cursor.fetchone() is not None
```

#### routes/member_routes.py — 路由层（只处理HTTP请求/响应）

```python
"""
路由层：只负责解析HTTP请求、调用Service、返回HTTP响应。
不做业务判断，不写SQL。
"""
from flask import Blueprint, request, jsonify, session, render_template
from services.member_service import get_member, search_members, create_member

member_bp = Blueprint('members', __name__)
conn = None   # 数据库连接（实际项目中通过依赖注入获取）

@member_bp.route('/members/<int:member_id>')
def view_member(member_id):
    """查看成员详情"""
    current_user_id = session.get('user_id')
    member, error = get_member(conn, member_id, current_user_id)
    if error:
        return jsonify({'error': error}), 403
    return render_template('member_detail.html', member=member)

@member_bp.route('/members/search')
def search():
    """模糊搜索成员"""
    keyword = request.args.get('keyword', '')
    genealogy_id = request.args.get('genealogy_id', type=int)
    current_user_id = session.get('user_id')
    results, error = search_members(conn, genealogy_id, keyword, current_user_id)
    return jsonify(results)

@member_bp.route('/members', methods=['POST'])
def add_member():
    """添加新成员"""
    data = request.get_json()
    current_user_id = session.get('user_id')
    result, error = create_member(
        conn,
        data['genealogy_id'],
        data['name'],
        data['gender'],
        data.get('birth_year'),
        current_user_id
    )
    if error:
        return jsonify({'error': error}), 400
    return jsonify(result), 201
```

### 7.4 各层职责对照表

| 层 | 目录 | 职责 | 禁止做的事 |
|---|------|------|-----------|
| **数据模型** | `models/` | 定义数据结构、属性、辅助方法 | 不写SQL、不做业务判断 |
| **数据访问** | `dao/` | 封装SQL语句、执行CRUD | 不做业务校验、不做权限判断 |
| **业务逻辑** | `services/` | 业务规则、权限控制、流程编排 | 不直接写SQL（调用DAO） |
| **路由** | `routes/` | 解析HTTP请求、调用Service、返回响应 | 不写SQL、不做复杂业务逻辑 |

### 7.5 为什么要这样分层？

1. **可维护性**：修改SQL只改DAO层，修改业务规则只改Service层，互不影响
2. **可测试性**：可以单独测试Service层（mock DAO层），不需要启动Web服务器
3. **可复用性**：同一个DAO层可以被不同的Service调用（如Web API和命令行工具）
4. **团队协作**：前端开发者改routes/templates，后端开发者改services/dao，互不冲突
